> **작성일**: 2026년 4월 16일 · **기준 버전**: OpenClaw 2026.4.14 (323493f)

## Executive Summary

OpenClaw는 2025년 11월 출시 후 6개월 만에 GitHub 스타 347,000개를 달성한 오픈소스 AI 에이전트 플랫폼이다. 그러나 급속한 성장과 함께 심각한 보안 문제들이 발견되었으며, 2026년 2월부터 4월까지 총 **138개의 CVE**가 보고되었다. 이 중 7개는 Critical, 49개는 High 심각도로 분류되었다.

2026.4.14 버전(323493f)은 이전 버전들의 보안 패치를 포함하며, 특히 2026.3.23부터 4.12까지의 대규모 보안 강화 작업의 혜택을 받는다. 이 보고서는 현재 버전의 보안 상태를 냉정하게 평가하고, 배포 방식별 위험을 비교 분석하며, 실용적인 보안 가이드라인을 제시한다.

---

## 1. 이전 버전 대비 보안 개선 사항

2026.3.23부터 4.12까지의 보안 업그레이드는 OpenClaw 프로젝트 역사상 가장 길고 기술적으로 밀도 높은 보안 패치 블록으로 평가받고 있다. 이 기간 동안 권한 격리(privilege containment), 워크스페이스 무결성(workspace integrity), 에코시스템 인증(ecosystem authorization), 크로스 컴포넌트 신뢰(cross-component trust) 전반에 걸친 수십 개의 패치가 적용되었다.

### 1.1 실행 환경 및 샌드박스 강화

- **호스트 환경 변수 살균(Host Environment Sanitization)**: HTTP/HTTPS 프록시, TLS 설정, Docker(DOCKER_HOST), Python 패키지 인덱스(PIP_INDEX_URL), Java(MAVEN_OPTS), Rust/Cargo, Git(GIT_EXEC_PATH), Kubernetes(KUBECONFIG), 클라우드 자격 증명(AWS_*, AZURE_*, GCLOUD_*) 등 위험한 환경 변수 주입을 차단
- **셸 래퍼 방어(Shell Wrapper Defenses)**: busybox와 toybox를 안전 바이너리 허용 목록에서 제거하여 내부 서브커맨드 호출을 통한 허용 목록 우회를 차단
- **승인 페일세이프(Approval Failsafes)**: 승인자 목록이 비어 있을 때 fail-closed 동작을 강제하여 의도치 않은 권한 부여를 방지
- **아카이브 트래버설 방어(Archive Traversal Hardening)**: TAR/ZIP 추출 시 symlink 탈출 공격을 차단하고, 워크스페이스 .env 파일이 핵심 런타임 제어를 덮어쓰지 못하도록 차단

### 1.2 네트워크 및 SSRF 방어

- 브라우저 내비게이션 후 차단된 대상 재검사(redirect safety re-check)로 SSRF 우회 차단
- 크로스 오리진 요청 시 본문/헤더 제거(stripping)로 페이로드 피벗 방지
- basic-ftp CRLF 인젝션 패치(4.9 버전)로 파일명을 통한 커맨드 인젝션 차단
- 엄격한 SSRF 기본값, 호스트명 화이트리스트, 서브프레임 및 Docker CDP 오리진 범위 적용

### 1.3 인증 및 권한 관리

- WebSocket/디바이스 세션 토큰 로테이션 시 무효화 처리
- 원격 노드 실행 이벤트를 비신뢰(untrusted)로 표시하고 출력 살균으로 프롬프트 인젝션 방지
- Skills 경로 검증에 realpath() 사용하여 symlink 기반 탈출 차단
- Owner-only 도구를 HTTP invoke 경로에서 차단

### 1.4 아키텍처 진화: Phase 9 성숙도

이 버전들의 보안 개선은 단순히 외부 위협을 방어하는 것을 넘어, **자체 컴포넌트를 체계적으로 불신하는(systematically distrusting its own components)** Phase 9 성숙도에 도달했다. 샌드박스, 플러그인, 브라우저, 원격 노드 간 권한 피벗을 방지하는 방어 종심(defense-in-depth) 아키텍처로 전환한 것은 의미 있는 진전이다.

### 2.2 여전히 존재하는 구조적 위험

패치로 해결된 CVE와 별개로, OpenClaw의 아키텍처 자체에서 비롯되는 구조적 위험이 존재한다:

1. **공급망 공격(Supply Chain Attacks)**: ClawHub 마켓플레이스에서 2,857개 스킬 중 341개(12%)가 악성 스킬로 확인되었으며, 키로거 및 Atomic Stealer 맬웨어를 배포했다. 스킬 등록 시 강력한 신원 확인이 부재한 점은 근본적 구조적 문제이다.
2. **노출된 인스턴스**: Censys 기준 21,639개의 인스턴스가 인터넷에 공개적으로 노출되어 있으며, 잘못 설정된 시스템에서 API 키, OAuth 토큰, 평문 자격 증명이 유출되고 있다.
3. **영구 메모리(Persistent Memory) 위험**: 에이전트가 세션 간 컨텍스트를 유지하므로, 에이전트 자체가 침해될 경우 접근 컨텍스트가 장기간 노출된다.
4. **기업 SaaS 연동 위험**: 직원들이 보안팀의 가시성 없이 기업 SaaS 앱에 OpenClaw를 연결할 경우, 이메일, 캘린더, 문서, OAuth 토큰 등에 대한 측면 이동(lateral movement) 경로가 열린다.

## 3. 보안 위험 강조의 필요성 검토

"많은 사용자가 사용하고 있으니 굳이 보안을 강조할 필요가 있느냐"라는 질문에 대해 냉정하게 답하면, 답은 **"네, 반드시 필요하다"**이다.

### 3.1 사용자 수가 많을수록 보안이 더 중요한 이유

1. **공격 표면적(Attack Surface) 확대**: 347,000 GitHub 스타와 수만 개의 활성 배포는 공격자에게 매력적인 타겟이다. 실제로 82개국에서 135,000개 이상의 노출된 인스턴스가 발견되었다.
2. **야생 악용 사례 존재**: CVE-2026-25253(ClawBleed)은 실제로 야생에서 악용된 것이 확인된 유일한 CVE이다. 이론적 위험이 아닌 현실의 위협이다.
3. **제3자 에코시스템 침해**: Moltbook의 보안 미비 데이터베이스에서 35,000개 이메일 주소와 150만 개 에이전트 API 토큰이 유출되었다.
4. **오픈소스 = 자동 안전이 아님**: 오픈소스는 코드의 투명성을 보장하지만, 보안을 자동으로 보장하지는 않는다. 오히려 코드가 공개되어 있으므로 공격자도 취약점을 분석할 수 있다.

### 3.2 균형 잡힌 시각

그러나 공포를 조장할 필요는 없다. 2026.4.14는 알려진 모든 치명적 CVE를 패치하였으며, 3월 이후 보안 아키텍처가 크게 개선되었다. 핵심은:

- 항상 최신 버전을 유지할 것
- 기본 설정을 그대로 사용하지 말고, 배포 환경에 맞게 보안 설정을 강화할 것
- 스킬/플러그인 설치 시 충분한 검증을 수행할 것

| 자격증명 보호 | ✖ 위험 | ○ 보통 | ○ 보통 | ● 안전 | ● 안전 |

### 4.1 방식별 상세 분석

**호스트 직접 설치 (D→B 등급, 조건부)**

OpenClaw가 호스트 시스템에서 직접 실행되며, 파일 접근과 명령 실행 권한이 호스트와 공유된다. CVE-2026-33579와 같은 권한 상승 취약점이 발생할 경우 호스트 전체가 위험에 노출된다. 다만 전용 서버에서 전용 Linux 사용자 계정으로 운영하고, 파일 권한 제한, Gateway를 127.0.0.1에만 바인딩, UFW 방화벽, Tailscale VPN 접근 제어를 적용하면 실무적으로 수용 가능한 수준의 보안을 확보할 수 있다. 많은 헤비 유저들이 이 방식을 채택하고 있다.

**Docker 컨테이너 (C 등급)**

컨테이너 격리가 기본 보호를 제공하지만, 포트가 외부에 노출될 경우 CVE-2026-25253과 같은 원격 공격에 취약하다. CVE-2026-32038처럼 Docker 네트워크 격리 자체를 우회하는 취약점도 발견된 바 있어 컨테이너만으로는 충분하지 않다. 또한 macOS에서는 Linux VM 계층으로 인한 성능 저하, MCP 서버 연동 시 호스트-컨테이너 간 네트워크 브릿지 설정 필요, 브라우저 자동화 도구와의 통신 지연 등 실무적 마찰이 발생한다.

**Docker + 역방향 프록시 + HTTPS (B 등급)**

역방향 프록시로 직접적인 포트 노출을 방지하고, HTTPS를 통해 전송 중 데이터를 보호한다. IP 제한을 추가하면 외부 공격 표면이 크게 줄어든다. 다만 공급망 공격과 스킬 보안은 별도로 관리해야 한다.

**클라우드 관리형 서비스 (A 등급)**

Blink Claw($45/월), xCloud($24/월) 등의 관리형 서비스는 보안 패치, SSL 갱신, 백업, 모니터링을 자동화한다. 운영 부담이 가장 낮으며, 보안 업데이트 지연으로 인한 위험이 최소화된다. 다만 데이터가 외부 서버에 전송되므로 민감한 데이터 처리 시 주의가 필요하다.

**VPN + Docker 조합 (A 등급)**

기업 환경에서 가장 권장되는 자체 호스팅 방식이다. VPN으로 네트워크 접근을 제한하고 Docker로 프로세스를 격리하면 외부 공격 표면이 사실상 제거된다. 데이터 주권도 유지된다.

### 4.2 Docker 없는 보안 확보: 실무적 접근법

조사 결과, 실제 헤비 유저 커뮤니티에서는 전용 서버를 구축하되 그 내부에서 Docker를 사용하지 않는 패턴이 상당히 보편적이다. 이는 Docker가 가져오는 실무적 마찰 때문이다: macOS에서의 Linux VM 오버헤드로 인한 성능 저하, MCP 서버와 브라우저 자동화 도구(Chrome DevTools MCP 등)의 호스트-컨테이너 간 통신 복잡성, 파일시스템 마운트 지연, 외부 도구와의 연계성 약화 등이 대표적이다.

Hacker News 커뮤니티의 합의도 주목할 만하다: **"샌드박스만으로는 OpenClaw를 보호할 수 없다."** 에이전트가 유용한 작업을 수행하려면 자격 증명과 네트워크 접근이 필요하므로, Docker 격리만으로는 근본적 위협을 해결하지 못한다. 오히려 중요한 것은 에이전트를 "신입 직원처럼" 취급하는 것이다: 별도의 기기, 제한된 계정, 명시적 승인 워크플로우.

**Docker 없이 동등한 보안을 확보하는 실무 체크리스트:**

- **전용 Linux 사용자 계정**: OpenClaw 전용 사용자를 생성하고, 해당 사용자로만 프로세스를 실행. 홈 디렉토리에 설정을 격리하고 최소 권한만 부여
- **Gateway 로컬 바인딩**: `gateway.bind`를 반드시 `127.0.0.1`로 설정. 절대 `0.0.0.0`에 바인딩하지 말 것. 원격 접근은 Tailscale 또는 SSH 터널을 통해서만 허용
- **도구 허용 목록(Tool Allowlist)**: `tools.exec.safeBins`에 필요한 바이너리만 명시적으로 등록. 거부 목록이 아닌 허용 목록 방식 사용. `tools.fs.workspaceOnly: true`로 파일 접근 범위를 제한
- **방화벽(UFW) + VPN(Tailscale)**: UFW로 기본 인바운드를 차단하고, SSH와 Tailscale만 허용. OpenClaw 포트는 외부에 절대 노출하지 않음
- **파일 권한 강화**: 민감한 설정 파일은 600, 디렉토리는 700으로 설정. 자격 증명은 클라우드 동기화 폴더에 절대 저장하지 않음
- **Human-in-the-Loop**: `tools.exec.ask: "always"`를 설정하여 상태 변경 작업(파일 삭제, 네트워크 요청, sudo 등)에 수동 승인을 강제
- **정기 보안 감사**: `openclaw security audit --fix`를 주기적으로 실행하여 설정 미비를 자동 점검. `--deep` 옵션으로 심층 감사를 수행
- **전용 격리 계정 사용**: 주요 이메일, 뱅킹, 패스워드 매니저, 업무 계정을 절대 OpenClaw에 연결하지 말 것. OpenClaw 전용으로 새로 생성한 격리 계정만 사용

> **핵심 결론**: 보안의 핵심은 "Docker 사용 여부"가 아니라 **"격리 원칙의 일관된 적용"**이다. Docker는 격리를 달성하는 하나의 수단이지 유일한 수단이 아니다. 전용 서버 + 전용 사용자 + 도구 허용 목록 + 네트워크 제어 + Human-in-the-Loop의 조합은 Docker 샌드박스와 동등하거나 그 이상의 실무적 보안을 제공한다.

## 5. 안전한 사용을 위한 보안 가이드라인

### 5.1 필수 조치 (Critical)

1. **항상 최신 버전 유지**: 2026.4.5 미만 버전은 취약한 것으로 간주해야 한다. 자동 업데이트 메커니즘을 구축할 것.
2. **실행 환경 격리 (Docker 또는 전용 사용자 계정)**: Docker 컨테이너가 이상적이지만, 실무에서는 MCP 도구 연동 제약, macOS에서의 VM 오버헤드, 설정 복잡도 증가 등으로 인해 Docker 없이 운영하는 사례가 많다. Docker를 사용하지 않을 경우, 전용 Linux 사용자 계정 생성, 파일 권한 600/700 설정, `tools.fs.workspaceOnly: true` 적용, `tools.exec.safeBins` 허용 목록 운영, `openclaw security audit --fix` 정기 실행 등으로 동등한 수준의 격리를 확보할 것. 핵심은 Docker 자체가 아니라 "격리 원칙의 실현"이다.
3. **외부 네트워크 노출 금지**: OpenClaw 포트를 인터넷에 직접 노출하지 말 것. VPN 또는 역방향 프록시 + IP 제한을 사용할 것.
4. **모든 노출된 자격 증명 즉시 폐기**: 이전 버전에서 노출되었을 수 있는 API 키, OAuth 토큰, 자격 증명을 모두 로테이션할 것.

### 5.2 권장 조치 (High)

1. **스킬/플러그인 검증 프로세스 수립**: ClawHub에서 스킬 설치 전 제작자 확인, 코드 리뷰, 권한 범위 검토를 수행. 전체 레지스트리의 12%가 악성이었던 사례를 기억할 것.
2. **최소 권한 원칙 적용**: OpenClaw 에이전트에 필요한 최소한의 파일 접근과 명령 실행 권한만 부여. 워크스페이스를 특정 디렉토리로 제한.
3. **네트워크 세분화**: OpenClaw 컨테이너를 별도의 Docker 네트워크에 배치하고, 필요한 외부 접근만 화이트리스트로 허용.
4. **모니터링 및 로깅**: OpenClaw 에이전트의 모든 활동을 로깅하고, 비정상적인 패턴(비정상 파일 접근, 예상치 못한 네트워크 요청)을 감지하는 알림을 설정.

### 5.3 기업 환경 추가 조치

1. **중앙 관리 정책 수립**: 직원들이 개별적으로 OpenClaw를 기업 SaaS에 연결하지 못하도록 정책을 수립하고, 보안팀의 검토를 거치도록 할 것.
2. **API 키 정기 로테이션**: LLM 프로바이더 API 키를 주기적으로 로테이션하고, 환경 변수를 통해 주입하지 말고 비밀 관리 도구(Vault 등)를 사용.
3. **정기 보안 감사**: OpenClaw 배포 환경에 대한 정기적인 보안 감사를 실시하고, 새로운 CVE 발표 시 즉시 대응할 수 있는 프로세스를 마련.

## 6. 결론

OpenClaw 2026.4.14(323493f)는 이전 버전 대비 보안 측면에서 상당한 진전을 이룬 버전이다. 특히 2026.3.23~4.12 기간의 보안 강화는 프로젝트 역사상 가장 대규모이며, Phase 9 성숙도의 방어 종심 아키텍처로 전환한 것은 의미 있는 발전이다.

그러나 5개월간 138개 CVE라는 숫자는 플랫폼의 보안 성숙도가 아직 성장 속도를 따라가고 있음을 보여준다. 공급망 공격, 기업 SaaS 연동 위험, 영구 메모리 침해 등의 구조적 위험은 패치만으로 해결되지 않으며, 운영 측의 보안 실천이 필수적이다.

> **최종 권고**: OpenClaw는 강력한 도구이지만, "설치하면 끝"이 아니라 "지속적으로 관리해야 하는" 도구이다. 격리 원칙(Docker 컨테이너 또는 전용 사용자+권한 제한), 네트워크 격리(VPN/Tailscale + 방화벽), 스킬 검증, 정기 업데이트를 기본으로 삼을 것. Docker가 이상적이지만 필수는 아니다. 전용 서버 + 전용 사용자 계정 + 도구 허용 목록 + 네트워크 제어의 조합으로도 실무적으로 충분한 보안을 확보할 수 있다.
