OpenClaw는 오스트리아 개발자 Peter Steinberger가 2025년 11월 "Clawdbot"이라는 이름으로 처음 공개한 오픈소스 개인 AI 에이전트입니다. Anthropic의 상표권 이의 제기 이후 2026년 1월 27일 "Moltbot"으로 개명했다가, 3일 뒤인 1월 30일 최종적으로 "OpenClaw"로 이름을 변경했습니다. 사용자의 로컬 머신(Mac, Windows, Linux)에서 24시간 상주하며, WhatsApp, Telegram, Slack, Discord 등 25개 이상의 메시징 플랫폼을 통해 이메일 관리, 캘린더 정리, 파일 조작, 웹 브라우징, 셸 명령 실행 등을 자율적으로 수행하는 "눈과 손이 달린" AI 비서입니다.

NanoClaw는 이스라엘 개발자 Gavriel Cohen이 OpenClaw의 보안 문제를 해결하기 위해 만든 경량 대안 AI 에이전트로, 2026년 1월 31일 MIT 라이선스로 공개되었습니다. 핵심 코드를 약 700줄의 TypeScript로 압축하고, 모든 에이전트 세션을 격리된 Docker 컨테이너 안에서 실행하는 "보안 우선" 설계 철학을 채택했습니다. Anthropic의 Claude Agent SDK 위에서 동작하며, 2026년 3월 Docker와 공식 파트너십을 체결했습니다.

## 프로젝트 규모 및 성장

| 지표 | OpenClaw | NanoClaw |
|------|----------|----------|
| **GitHub Stars** | ~354,000+ (2026년 3월 기준) | ~11,000+ (공개 1개월 내) |
| **Forks** | 47,700+ | 다수의 포크 활발 |
| **Contributors** | 600~1,700명 | 소규모 핵심 기여자 |
| **Commits** | 10,000+ | 상대적으로 적음 |
| **코드 규모** | ~500,000줄, 53개 설정 파일, 70+ 의존성 | ~700줄 TypeScript, 15개 소스 파일 |
| **첫 공개일** | 2025년 11월 | 2026년 1월 31일 |
| **라이선스** | 오픈소스 | MIT 라이선스 |

OpenClaw는 3개월 만에 GitHub 역사상 가장 빠르게 성장한 프로젝트 중 하나가 되었으며, 글로벌 랭킹 6위(354K 스타)에 올랐습니다. NanoClaw는 공개 일주일 만에 7,000 스타를 넘기며 AI 에이전트 커뮤니티에서 큰 주목을 받았습니다.

## 기능성 비교

### 지원 메시징 채널

**OpenClaw (25개 이상):** WhatsApp, Telegram, Slack, Discord, Google Chat, Signal, iMessage, BlueBubbles, IRC, Microsoft Teams, Matrix, Feishu, LINE, Mattermost, Nextcloud Talk, Nostr, Synology Chat, Tlon, Twitch, Zalo, WeChat, QQ, WebChat 등

**NanoClaw (5~6개 기본, 스킬로 확장):** WhatsApp(기본), Telegram, Discord, Slack, Gmail, Signal (스킬을 통해 /add-telegram, /add-discord, /add-slack 등으로 추가)

OpenClaw가 채널 지원 범위에서 압도적으로 넓습니다.

### 핵심 기능

| 기능 | OpenClaw | NanoClaw |
|------|----------|----------|
| **웹 브라우징** | 내장 | 컨테이너 내 Chromium |
| **파일 읽기/쓰기** | 호스트 직접 접근 | 컨테이너 내 격리 |
| **셸 명령 실행** | 호스트에서 실행 | 컨테이너 내부에서 실행 |
| **웹 검색** | 내장 | Claude Code 내장 |
| **이메일 관리** | 지원 | Gmail 스킬 |
| **캘린더 관리** | 지원 | 제한적 |
| **음성 입출력** | macOS/iOS/Android 지원 | 미지원 |
| **웹 대시보드** | 지원 | 미지원 (최소 인터페이스) |
| **CLI 클라이언트** | 지원 | 미지원 |
| **모바일 앱** | iOS, Android 지원 | 미지원 |
| **라이브 캔버스** | 지원 | 미지원 |
| **예약 작업 (Cron)** | 지원 | 지원 (SQLite 기반) |
| **영구 메모리** | 지원 | SQLite 기반 지원 |
| **에이전트 스웜(위임)** | 제한적 | Claude Code 기반 지원 |

### LLM 제공자 지원

OpenClaw는 **모델 애그노스틱(model-agnostic)** 설계로, Anthropic Claude뿐 아니라 OpenAI, Google, 로컬 모델 등 다양한 LLM 제공자를 지원합니다. NanoClaw는 **Anthropic Claude 전용(Claude-centric)** 설계로, Claude Agent SDK 위에서만 동작하며 단일 제공자에 의존합니다.

### 스킬/플러그인 생태계

**OpenClaw:** ClawHub라는 커뮤니티 마켓플레이스를 운영하며, 2026년 2월 기준 13,729개의 커뮤니티 빌트 스킬이 등록되어 있습니다. 누구나 GitHub 계정으로 스킬을 업로드할 수 있으나, 이로 인해 보안 문제가 발생하기도 했습니다(후술).

**NanoClaw:** 중앙 마켓플레이스 대신 "포크 기반 커스터마이징" 철학을 채택했습니다. 코어 코드베이스에 기능을 추가하기보다, 사용자가 저장소를 포크해서 자신만의 스킬을 소스코드 수준에서 직접 추가하는 방식입니다. 따라서 스킬 수는 적지만 각 스킬의 신뢰도를 사용자가 직접 통제합니다.

## 보안성 비교

### 보안 아키텍처

**OpenClaw의 접근 방식 — 애플리케이션 레벨 보안:**
OpenClaw는 게이트웨이 인증, 디바이스 페어링, 허용 목록(allowlist), 도구 정책(tool policy), 실행 승인 워크플로우, 샌드박싱 권고사항 등 다층적 보안 메커니즘을 제공합니다. 그러나 이 모든 것이 "권고사항"이나 "설정"에 의존하며, 기본 설정(default)이 보안적으로 취약한 경우가 많습니다.

**NanoClaw의 접근 방식 — OS 레벨 컨테이너 격리:**
모든 에이전트 세션이 자체 Docker 컨테이너(또는 macOS의 Apple Container) 안에서 실행됩니다. 각 컨테이너는 고유한 파일시스템, IPC 네임스페이스, 프로세스 공간을 가지며, 호스트 시스템에는 접근할 수 없습니다. 쓰기 작업은 기본적으로 차단되어 있고, 명시적 허가가 있어야만 활성화됩니다.

### OpenClaw의 보안 이슈 (심각)

OpenClaw는 2026년 들어 심각한 보안 위기를 겪고 있습니다:

- **CVE 누적:** 2026년 4월 6일 기준 63일 동안 138개의 CVE가 발견됨 (하루 평균 2.2건)
- **심각도:** 전체 CVE의 41%가 High 또는 Critical 등급
- **주요 취약점:** 명령 인젝션(CVE-2026-24763), SSRF(CVE-2026-26322), 경로 탐색(CVE-2026-26329), 프롬프트 인젝션 기반 코드 실행(CVE-2026-30741)
- **2026년 3월 18~21일:** 4일간 9개 CVE 동시 공개, 그 중 1건은 CVSS 9.9/10 (Critical)
- **노출 규모:** 2026년 2월 Shodan 스캔 결과 42,000개 이상의 OpenClaw 인스턴스가 공개 인터넷에 노출. 이 중 63%가 게이트웨이 인증 비활성화 상태, 28%가 패치 이전 버전
- **공급망 공격:** ClawHub에 등록된 10,700개 스킬 중 820개 이상이 악성 스킬로 확인됨 (사전 코드 리뷰 부재)

### NanoClaw의 보안 강점

- **공격 표면 최소화:** 코드 전체가 ~700줄이므로, 한 사람이 약 8분 만에 전체 감사(audit) 가능
- **기본 격리:** 컨테이너 기본 실행으로 호스트 파일시스템·네트워크·다른 컨테이너 접근 차단
- **권한 명시:** 쓰기, 메시지 전송, 코드 실행 등 모든 위험 작업에 명시적 허가(explicit allow grant) 필요
- **의존성 최소화:** 70+ 의존성 vs 소수 의존성 → 공급망 공격 표면 대폭 축소
- **Docker 파트너십:** 2026년 3월 Docker MicroVM 기반 샌드박스와 통합, 더 강력한 격리 환경 제공

### 보안 비교 요약

| 항목 | OpenClaw | NanoClaw |
|------|----------|----------|
| **격리 수준** | 애플리케이션 레벨 (allowlist, 페어링) | OS 레벨 (Docker 컨테이너) |
| **기본 보안 상태** | 다수 설정이 기본 비활성화 | 기본적으로 격리·읽기 전용 |
| **알려진 CVE** | 138개+ (2026.4 기준) | 보고된 CVE 없음 |
| **코드 감사 용이성** | ~500,000줄 (감사 어려움) | ~700줄 (8분 감사 가능) |
| **공급망 리스크** | 820+ 악성 스킬 발견 | 포크 기반, 중앙 마켓플레이스 없음 |
| **인증 기본값** | 다수 인스턴스 미인증 노출 | 컨테이너 격리가 기본 |

## 활용성 비교

### 설치 및 설정

**OpenClaw:** 다양한 설치 옵션 제공 (npm, Docker, 소스 빌드), 풍부한 설정 옵션과 웹 대시보드. 그러나 53개 설정 파일과 복잡한 옵션으로 초보자에게 진입 장벽이 높을 수 있습니다.

**NanoClaw:** Docker만 설치되어 있으면 단일 명령으로 실행 가능. 15개 소스 파일로 전체 구조 파악이 쉽고, 포크 후 커스터마이징하는 단순한 워크플로우. 그러나 Docker에 익숙하지 않은 사용자에게는 오히려 부담이 될 수 있습니다.

### 적합한 사용 시나리오

**OpenClaw가 더 적합한 경우:**
- 25개 이상의 메시징 플랫폼 중 다양한 채널을 동시에 사용해야 할 때
- 모바일(iOS/Android)에서도 에이전트에 접근해야 할 때
- Claude 외에 다른 LLM도 사용하고 싶을 때
- 웹 대시보드, CLI 등 풍부한 관리 도구가 필요할 때
- ClawHub 생태계의 13,000+ 스킬을 활용하고 싶을 때
- 대규모 팀이나 조직에서 다수의 노드를 운영해야 할 때

**NanoClaw가 더 적합한 경우:**
- 보안이 최우선 과제인 환경 (금융, 의료, 법률 등)
- 코드 감사 가능성(auditability)이 중요한 조직
- 에이전트의 모든 동작을 완전히 이해하고 통제하고 싶을 때
- Claude를 주 LLM으로 사용하는 경우
- 가벼운 개인용 AI 비서가 필요한 경우
- OpenClaw의 CVE 이력이 조직 정책상 허용되지 않는 경우

### 성능

| 지표 | OpenClaw | NanoClaw |
|------|----------|----------|
| **시작 시간** | ~12ms (프로세스 포크) | ~95ms (Docker 컨테이너 초기화) |
| **리소스 사용** | 상대적으로 높음 (게이트웨이 + 다중 서비스) | 낮음 (단일 Node.js 프로세스 + 컨테이너) |

OpenClaw가 시작 속도에서 약 8배 빠르지만, NanoClaw의 95ms도 실사용에서 체감되지 않는 수준입니다.

## 종합 평가

### OpenClaw의 강점
1. 압도적인 커뮤니티 규모와 생태계 (354K+ 스타, 13,000+ 스킬)
2. 25+ 메시징 플랫폼 지원으로 최고의 범용성
3. 모델 애그노스틱 설계로 LLM 선택의 자유
4. 모바일 앱, 웹 대시보드, 음성 등 풍부한 인터페이스
5. 활발한 개발과 빠른 기능 추가

### OpenClaw의 약점
1. 심각한 보안 이슈 (138+ CVE, 악성 스킬, 노출된 인스턴스)
2. 복잡한 코드베이스로 감사 및 이해 어려움
3. 기본 보안 설정이 느슨해 사용자 실수에 취약
4. 공급망 공격에 노출된 스킬 마켓플레이스

### NanoClaw의 강점
1. OS 레벨 컨테이너 격리로 근본적으로 안전한 설계
2. 700줄의 간결한 코드로 완벽한 감사 가능성
3. Docker 공식 파트너십으로 엔터프라이즈 신뢰 확보
4. 기본 보안이 강력 (읽기 전용, 명시적 허가 필요)
5. Claude Agent SDK와의 긴밀한 통합

### NanoClaw의 약점
1. 지원 채널이 제한적 (기본 5~6개)
2. Claude 전용으로 LLM 선택지 없음
3. 웹 대시보드, 모바일 앱 등 편의 기능 부재
4. 상대적으로 작은 커뮤니티와 스킬 생태계
5. Docker 의존성 (Docker에 익숙하지 않은 사용자에게 허들)

## 선택 가이드

**"기능이 풍부한 범용 AI 비서"가 필요하다면 → OpenClaw** (단, 보안 설정을 반드시 강화할 것)

**"보안이 최우선인 신뢰할 수 있는 AI 비서"가 필요하다면 → NanoClaw**

두 프로젝트는 경쟁 관계라기보다 서로 다른 설계 철학의 산물입니다. OpenClaw는 "모든 것을 할 수 있는 플랫폼"을 지향하고, NanoClaw는 "안전하게 핵심만 하는 도구"를 지향합니다. 조직의 보안 요구사항, 사용 채널, LLM 선호도, 그리고 기술 역량에 따라 선택이 달라질 것입니다.
