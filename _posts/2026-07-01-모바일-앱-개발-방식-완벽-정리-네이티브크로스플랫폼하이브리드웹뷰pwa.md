---

## 1. 왜 용어가 헷갈리는가 — 분류의 두 축

모바일 앱 개발 방식의 용어가 혼용되는 이유는 서로 다른 **두 가지 기준**이 섞여 있기 때문이다.

- **축 1 — UI 렌더링 방식**: 네이티브 위젯 / 자체 렌더링 엔진 / 웹(HTML·CSS)
- **축 2 — 코드 공유(멀티플랫폼)**: 플랫폼별 개별 개발 / 하나의 코드로 양쪽

이 두 축을 구분하지 않으면 "RN을 하이브리드라 부르는" 식의 부정확한 표현이 생긴다.

---

## 2. 전체 분류 체계

렌더링 계열(네이티브 위젯 / 자체 엔진 / 웹)로 대분류하고, WebView 계열은 세부 하위 범주까지 전개한다.

| 대분류(렌더링 계열) | 세부 구분 · 정식 용어 | 자주 쓰는 표현 | 대표 기술 | UI 렌더링 |
|---|---|---|---|---|
| **네이티브 위젯** | ① 네이티브 · **Native App** | "순수 네이티브" | Kotlin·Java, Swift·Obj-C | OS 네이티브 위젯 |
| **네이티브 위젯** | ② 크로스플랫폼(네이티브) · **Cross-platform (Native)** | "크로스플랫폼" | React Native, .NET MAUI, KMP | 네이티브 위젯(브릿지) |
| **자체 렌더링** | ③ 크로스플랫폼(자체 엔진) · **Cross-platform** | "플러터 계열" | **Flutter** | 자체 엔진(Skia·Impeller) |
| **웹(WebView)·앱** | ④ 하이브리드 · **Hybrid App** | "코르도바/이오닉 앱" | **Cordova, Capacitor, Ionic** | WebView(웹) + 플러그인 |
| **웹(WebView)·앱** | ⑤-a 순수 웹뷰 래퍼 · **Pure WebView Wrapper** | "웹뷰 앱", "래퍼 앱" | 네이티브 껍데기 + 원격 URL | WebView(웹) |
| **웹(WebView)·앱** | ⑤-b 네이티브 셸 하이브리드 · **Bridged Wrapper / Native-Shell Hybrid** | "브릿지 기반 하이브리드", "부분 웹뷰" | 웹뷰 + 커스텀 네이티브 브릿지 | WebView(웹) + 네이티브 기능 |
| **웹(브라우저)** | ⑥ 웹앱 · **Web App** | "웹앱" | 브라우저 실행 | 웹 |
| **웹(브라우저)** | ⑦ 반응형 웹 · **Responsive Web** | "반응형 웹" | 미디어쿼리 대응 웹 | 웹 |
| **웹(브라우저)** | ⑧ 프로그레시브 웹앱 · **PWA** | "PWA" | Service Worker·Manifest | 웹(설치·오프라인) |

> **⑤ 웹뷰 래퍼는 브릿지 유무로 두 갈래**로 나뉜다. 브릿지 없이 원격 웹만 감싸면 ⑤-a(순수 래퍼), 푸시·로그인·IAP 등 핵심 기능을 커스텀 네이티브 브릿지로 붙이면 ⑤-b(네이티브 셸 하이브리드)다.

### 계층도

```
모바일 앱
├─ [네이티브 위젯 렌더링]
│   ├─ ① Native (네이티브)                  … Kotlin / Swift
│   └─ ② Cross-platform (네이티브 위젯)      … React Native, MAUI
├─ [자체 렌더링]
│   └─ ③ Cross-platform (자체 엔진)          … Flutter
├─ [웹(WebView) 렌더링 · 앱]
│   ├─ ④ Hybrid (하이브리드)                … Cordova, Capacitor, Ionic  (웹으로 앱 전체 제작)
│   └─ ⑤ WebView Wrapper (웹뷰 래퍼)
│       ├─ ⑤-a 순수 웹뷰 래퍼               … 원격 URL 로드, 브릿지 최소/없음
│       └─ ⑤-b 네이티브 셸 하이브리드         … 웹뷰 + 커스텀 네이티브 브릿지
│              (= 브릿지 기반 하이브리드)         (푸시·로그인·IAP만 네이티브)
└─ [웹(브라우저)]
    ├─ ⑥ Web App (웹앱)
    ├─ ⑦ Responsive Web (반응형 웹)
    └─ ⑧ PWA (프로그레시브 웹앱)
```

### ④ 하이브리드 vs ⑤-b 네이티브 셸 하이브리드 — 무엇이 다른가

기능적으로 수렴하지만 **출발점과 도구**가 다르다.

| 기준 | ④ 하이브리드(Capacitor/Cordova) | ⑤-b 네이티브 셸 하이브리드 |
|---|---|---|
| 출발점 | "웹 기술로 **앱을 새로 만든다**" | "**기존 웹을 감싸고** 핵심 기능만 네이티브" |
| 웹 자산 | 앱에 로컬 번들이 기본 | 원격 URL 로드가 기본 |
| 네이티브 접근 | **프레임워크가 플러그인/브릿지 표준 제공** | **브릿지를 직접 구현**(또는 최소 플러그인) |
| 대표 도구 | Capacitor, Cordova, Ionic | 자체 네이티브 셸 + JS 브릿지 |

> 두 방식은 브릿지를 늘릴수록 서로 가까워진다. 실무에서는 "Capacitor로 시작하면 ④, 기존 웹앱을 감싸며 브릿지를 붙여 나가면 ⑤-b"로 이해하면 된다.

## 3. 방식별 상세

### ① 네이티브 (Native)
- 플랫폼 전용 언어·SDK로 개발: Android=Kotlin/Java, iOS=Swift/Objective-C.
- **장점**: 최고 성능·UX, 최신 OS 기능 즉시 사용.
- **단점**: iOS/Android 두 벌 개발 → 비용·인력 2배.

### ② 크로스플랫폼 — 네이티브 위젯 방식 (React Native 등)
- JS/TS 코드 → **브릿지(JSI)** 로 OS의 **진짜 네이티브 위젯**을 그림.
- 겉모습이 OS 네이티브에 가깝고, 실시간 오디오·캔버스 등 라이브러리 생태계 성숙.

### ③ 크로스플랫폼 — 자체 렌더링 방식 (Flutter)
- Dart 코드 → **자체 렌더링 엔진(Skia/Impeller)** 이 모든 픽셀을 직접 그림. 네이티브 위젯을 쓰지 않음.
- 일관된 UI·고성능 애니메이션에 강점.
- ②와 ③ 모두 "크로스플랫폼"이지만 **렌더링 원리가 근본적으로 다름**.

### ④ 하이브리드 (Hybrid)
- **웹 기술로 앱 전체를 만들고** 네이티브 컨테이너에 담음. 네이티브 기능은 **플러그인/브릿지**로 접근.
- 대표: Cordova(레거시), **Capacitor(현행 표준)**, Ionic.
- 웹 자산은 보통 앱에 **로컬 번들**(오프라인 기본).

### ⑤ 웹뷰 래퍼 (WebView Wrapper)
- **⑤-a 순수 웹뷰 래퍼**: 이미 존재하는 웹사이트/웹앱을 WebView로 감싸 스토어에 배포. 원격 URL 로드, 네이티브 연동 최소.
- **⑤-b 네이티브 셸 하이브리드(브릿지 기반 하이브리드)**: 웹뷰 + 푸시·로그인·IAP 등 핵심 기능만 커스텀 네이티브 브릿지로 처리.

### ⑥⑦⑧ 웹 계열
- **웹앱/반응형 웹**: 브라우저에서 실행. 앱 아님.
- **PWA**: Service Worker·Manifest로 설치·오프라인·푸시(제한적)를 지원하는 진화형 웹앱.

## 4. 하이브리드 vs 웹뷰 래퍼 — 심층 비교

둘 다 WebView로 화면을 그리지만, **"웹으로 앱을 만드느냐" vs "기존 웹을 껍데기로 감싸느냐"** 에서 갈린다.

### 4.1 본질 차이

| | 하이브리드 앱(④) | 웹뷰 래퍼 앱(⑤-a) |
|---|---|---|
| 대표 기술 | Cordova, Capacitor, Ionic | 네이티브/RN 껍데기 + `WebView`/`WKWebView` |
| 웹 자산 위치 | 앱에 **번들(로컬)** 기본 | **원격 URL 로드** 기본 |
| 네이티브 접근 | **플러그인/브릿지 풍부** | 거의 없음(있어도 최소) |
| 설계 의도 | "웹 기술로 만든 앱" | "웹사이트를 앱으로 배포" |
| 제작 비용 | 중 | 최소 |

### 4.2 항목별 장단점

| 평가 축 | 하이브리드(Cordova/Capacitor) | 웹뷰 래퍼 |
|---|---|---|
| 개발 속도/비용 | ✅ 웹 스택 재사용·플러그인 | ✅✅ 기존 웹 있으면 며칠이면 앱화 |
| 네이티브 기능 | ✅ 플러그인 생태계로 대부분 커버 | ❌ 기본 불가, 직접 브릿지 개발 필요 |
| 오프라인 지원 | ✅ 로컬 번들 | ❌ 원격 로드면 네트워크 필수 |
| 성능/UX | △ 최적화 여지 있음 | ❌ 로딩·제스처·전환 어색 |
| OTA 업데이트 | ✅ 웹 자산 교체 | ✅✅ 서버 웹만 바꾸면 끝 |
| 앱스토어 심사 | △ 기능 충분하면 통과 | ❌❌ **Apple 4.2 "minimum functionality" 리젝 위험** |
| 유지보수/재사용 | ✅ 웹·앱 코드 일원화 | ✅✅ 웹 하나로 web/app 동시 |
| 확장성 | ✅ 플러그인·네이티브 모듈로 확장 | ❌ 복잡해질수록 한계 |

### 4.3 핵심 트레이드오프
1. **앱스토어 심사**: 순수 웹뷰 래퍼는 Apple Guideline 4.2/4.2.3 리젝 위험이 크다. 네이티브 기능 1~2개를 넣어 "앱다움"을 확보하면 통과 → 이 순간 사실상 하이브리드(⑤-b)로 이동.
2. **OTA 업데이트**: 웹뷰 계열의 최대 강점. 단, Apple은 심사 우회로 핵심 기능을 원격 변경하는 것을 제한.
3. **성능·UX 천장**: WebView는 실시간 오디오·영상·캔버스·고빈도 인터랙션에 약함.

## 5. 네이티브 셸 + 브릿지 패턴 (실무의 스위트스팟)

"웹뷰 래퍼에 브릿지·플러그인을 커스텀 네이티브로 붙이면 장점이 된다"는 것은 정확하며, 이는 정식 패턴이다.

- 정식 명칭: **네이티브 셸 하이브리드 / 브릿지 기반 하이브리드**(= 2장 ⑤-b).
- **브릿지를 붙이는 순간 그것은 이미 순수 래퍼가 아니라 하이브리드**다. 래퍼↔하이브리드는 스펙트럼이며 브릿지 개수만큼 하이브리드로 이동한다.
- 국내 커머스·금융·슈퍼앱 다수가 이 구조(웹 콘텐츠 + 푸시·로그인·결제·생체인증만 네이티브).

### 5.1 브릿지로 처리하는 대표 기능과 정책 강제사항

| 기능 | 네이티브 처리 이유 | 정책/강제 사항 |
|---|---|---|
| **푸시 알림** | FCM/APNs는 본질적으로 네이티브. iOS 웹푸시 제약 큼 | 사실상 네이티브 강제 |
| **SNS 로그인**(카카오·네이버·애플·구글) | 네이티브 SDK가 UX·성공률 우수 | iOS는 타 소셜로그인 있으면 **Sign in with Apple 필수(4.8)** |
| **인앱 결제(디지털재화)** | 앱 내 디지털 구독 | **StoreKit/Play Billing 강제** — 웹결제 사용 시 리젝. 수수료 15~30% |
| 생체인증·보안저장 | Keychain/Keystore | 네이티브만 안전 |

> **인앱 결제는 선택이 아니라 스토어 정책상 네이티브 강제**다(디지털재화 기준). 이 기능들을 넣으면 덤으로 Apple 4.2 리젝 리스크도 사라진다.

### 5.2 브릿지 패턴의 함정
1. **"저렴·빠름" 장점 감소**: 커스텀 브릿지는 iOS/Android 네이티브 개발자를 요구하고 유지보수 레이어를 만든다.
2. **버전 스큐**: 웹은 OTA로 갱신되지만 **네이티브 브릿지는 앱 업데이트 전까지 고정**. 웹이 없는 브릿지 메서드를 호출하면 크래시 → **capability negotiation**(`bridge.has('iap')` 확인 후 호출)과 graceful degradation 필수.
3. **상태 동기화**: 네이티브 로그인 토큰을 WebView에 주입(쿠키/헤더/localStorage), 세션·로그아웃 동기화가 실제 공수.
4. **바퀴 재발명 여부**: 푸시·로그인·IAP는 Capacitor 플러그인으로 검증된 구현이 존재. 커스텀은 "플러그인이 못 주는 제어가 필요할 때"만.

### 5.3 스펙트럼

```
순수 웹뷰 래퍼 ──(브릿지 추가)──▶ 네이티브 셸 하이브리드 ──▶ 하이브리드 ──▶ 네이티브
  (⑤-a)                          (⑤-b)                    (④)
  최소비용                    푸시·로그인·IAP만 네이티브        점점 네이티브 비중↑
  4.2 리젝위험 큼                 4.2 통과, 실무 표준
```

## 6. Capacitor 심층 정리

### 6.1 개요
- **Ionic 팀이 만든 오픈소스 크로스플랫폼 런타임**. 웹 앱을 iOS·Android·웹으로 감싸고 네이티브 기능을 플러그인/브릿지로 노출.
- **Cordova의 사실상 후계자**. "네이티브 셸 + 브릿지 + 웹뷰" 구조를 표준화.
- **프레임워크 중립**: React·Vue·Angular·Svelte·바닐라 등 어떤 웹 스택과도 조합(Ionic UI 강제 아님).
- 2025년 주간 다운로드 약 93만(100만 근접) — 하이브리드 진영 실질 표준.

### 6.2 아키텍처

```
[웹 앱 (React/Vue 등)]
        │  Capacitor Bridge (JS ↔ Native)
        ▼
[네이티브 셸: WKWebView(iOS) / WebView(Android)]
        │  Plugins
        ▼
[네이티브 API: 카메라·푸시·생체인증·파일·결제…]
```
- 웹 자산은 앱에 **로컬 번들**(오프라인 기본). 원격 URL 로드도 가능.
- iOS/Android **네이티브 프로젝트를 소스로 소유**(Xcode/Android Studio로 직접 편집) → 커스텀 네이티브 코드 혼합이 쉬움. **Cordova와의 결정적 차이**.

### 6.3 Capacitor vs Cordova

| 항목 | Capacitor | Cordova(레거시) |
|---|---|---|
| 네이티브 프로젝트 | **소스로 소유·직접 편집** | 빌드 시 자동 생성(수정 어려움) |
| 커스텀 네이티브 코드 | ✅ 쉬움 | △ 훅/스크립트 우회 |
| 플러그인 모델 | 현대적(Swift/Kotlin) | 노후화 |
| PWA/웹 지원 | ✅ 1급 | 제한적 |
| 유지보수 활발도 | ✅ 활발 | 사실상 정체 |

→ **신규 프로젝트는 Capacitor**, Cordova는 레거시.

### 6.4 장점 / 단점

**장점**
- 웹 스택 재사용, web·iOS·Android 코드 일원화
- 네이티브 프로젝트 소유 → 푸시·로그인·IAP를 커스텀 네이티브로 자유롭게 구현
- OTA 업데이트(Capgo·Appflow 등) — 스토어 재심사 없이 웹 자산 갱신
- 웹 자산 로컬 번들로 오프라인 기본 지원
- 풍부한 플러그인(공식 + 커뮤니티 + Capawesome)
- 프레임워크 중립
- 네이티브 기능 탑재로 앱스토어 4.2 리스크 완화

**단점 / 한계**
- **WebView 성능 천장**: 실시간 오디오/영상·캔버스·고빈도 인터랙션 한계
- 커스텀 플러그인·브릿지 제작 시 결국 Swift/Kotlin 인력 필요
- 버전 스큐(웹 OTA ↔ 네이티브 브릿지 고정)
- UX가 완전 네이티브만은 못함(제스처·전환·키보드)
- iOS WKWebView 제약(쿠키·CORS·파일 접근)
- 최신 환경 요구(Capacitor 8: Node 22+, 최신 Xcode/Android SDK)

### 6.5 최신 동향 — Capacitor 8 (2025-12-08 릴리스)
- **SPM(Swift Package Manager) 기본화**: 신규 iOS 프로젝트는 CocoaPods → SPM(기존 프로젝트는 계속 동작).
- **SystemBars 코어 플러그인**: Android edge-to-edge(상태바/네비바 인셋) 자동 지원 + 세밀 제어 API. 기존 `@capacitor/status-bar`와 병행 가능.
- 플랫폼 현대화, 플러그인 config `getDouble`, CLI 실험적 `packageOptions` 등.
- **요구사항**: Node 22+.

### 6.6 언제 적합한가

| 상황 | 판단 |
|---|---|
| 기존 웹 자산 + 콘텐츠 중심 앱 | ✅ 최적 |
| 푸시·SNS로그인·IAP만 네이티브 | ✅ 플러그인/커스텀 브릿지로 커버 |
| web·app 코드 일원화·빠른 배포 | ✅ |
| 실시간 음성·영상·게임·고성능 UX | ❌ 네이티브/RN/Flutter |

| 흔한 표현 | 정식 용어 |
